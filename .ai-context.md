## Multi-Domain Routing

Same build is served on all three domains. Domain detected at **runtime** via `window.location.hostname`.

```
src/lib/portalConfig.js   ← single source of truth for domain logic
```

| Domain | Renders |
|---|---|
| `zaproc.co.uk` | Zaproc `LandingPage` |
| `rzglobalsolutions.co.uk` | `RZGBLandingPage` |
| `portal.zaproc.co.uk` | Full app (auth required) |

Key functions: `isLandingDomain()`, `isRZGBDomain()`, `isAnyLandingDomain()`, `getPortalUrl()`

`DomainGuard` component — redirects any non-root path on landing domains to `portal.zaproc.co.uk`.

## Authentication & Roles

Auth via Supabase. All data scoped by `workspace_id`.

| Role | Access |
|---|---|
| `super_admin` | Platform admin (`/platform-admin`) — full system |
| `customer_admin` | Workspace admin (`/control-centre`) |
| `client` | Client portal |
| `supplier` | Supplier portal |

Context providers (wrap entire app in `App.jsx`):
- `AuthContext` — current user, role, session
- `AdminContext` — admin-scoped data
- `SupplierContext` — supplier-scoped data
- `ClientContext` — client-scoped data
- `ThemeContext` — light/dark theme

## Known Issues / Watch Out For

- `DomainGuard` redirects ALL non-root paths on landing domains to portal
- Same Vite build serves all three domains — no per-domain builds
- `VITE_*` env vars are baked at build time, not runtime
- ErrorBoundary catches React render errors but NOT module-level crashes
